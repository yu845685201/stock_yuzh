# 股票静态分析系统产品设计文档

## 系统概述

作为专业的股票数据分析系统开发专家，我们将基于以下需求设计并实现一套完整的股票静态分析系统。该系统专注于A股市场的盘后数据分析，核心功能涵盖数据同步和股票分析两大模块。

## 核心设计原则

### 数据质量与真实性原则

#### 数据源可靠性原则
- **源数据可验证性**：所有数据必须来自经过严格验证的真实数据源
- **数据篡改零容忍**：杜绝任何形式的数据修改、伪造或模拟行为

#### 数据记录完整性原则
- **分级获取策略**：
  - 第一优先级：直接从官方数据源获取原始数据
  - 第二优先级：通过可靠计算模型推导相关数据
  - 第三优先级：无法获取则明确标注并记录具体原因

### 技术架构与工程原则

#### 技术栈统一性原则
- **开发语言与框架**：采用Python 3.8+作为后端主要开发语言，支持异步架构
- **数据库选型标准**：使用PostgreSQL 12+作为主数据库，扩展支持TimescaleDB时序数据处理
- **技术兼容性保证**：确保所有技术组件版本兼容，降低系统集成复杂度

#### 架构设计原则
- **接口标准化原则**：
  - 抽象数据源接口设计，支持热插拔机制
  - 统一数据存储接口标准，支持多种存储后端
  - 规范化分析处理接口，确保系统间协作效率
- **插件化架构原则**：采用模块化插件架构，新增功能无需修改核心代码
- **配置驱动原则**：通过统一配置文件管理数据源和分析策略，提升系统灵活性

#### 多数据源管理原则
- **数据源优先级策略**：
  - 主要数据源：pytdx（负责K线数据）、baostock（负责基本面数据）
  - 数据源协同机制：严格按照数据来源说明进行数据采集，确保数据一致性
  - 扩展接口规范：预留标准化数据源接入接口，支持未来数据源扩展

#### 系统模块化设计原则
- **模块独立性原则**：确保各功能模块高内聚、低耦合
- **接口清晰定义**：明确模块间接口标准，降低系统集成复杂度

### 开发与维护原则

#### 代码质量与可维护性原则
- **代码规范一致性**：遵循统一的编码规范和最佳实践
- **文档完整性要求**：确保技术文档、API文档和用户文档的完整性和时效性

## 核心功能模块

### 数据同步功能

#### 功能特性

- **手动触发同步**：用户可通过命令行或API手动触发数据同步
- **日期范围选择**：支持指定起始和结束日期进行数据同步
- **多维度数据采集**：支持按市场（沪市、深证、北交所）和指定股票编码进行数据采集
- **双存储策略**：数据同时保存为CSV文件和数据库记录，CSV文件和数据库记录结构确保完全一致
- **并发控制机制**：采用多线程进行数据采集
- **进度实时监控**：实时显示同步进度和状态
- **采集结果反馈**：数据采集结束时显示采集总量、成功数量、失败数量及具体失败股票编码

#### 同步流程优化

```
开始同步
    ↓
参数验证与预处理
    ↓
数据源连接与验证
    ↓
数据分片与任务分配
    ↓
并行数据采集（多线程/协程）
    ↓
数据清洗与转换
    ↓
数据质量检查
    ↓
双存储写入（CSV + Database）
    ↓
数据完整性校验
    ↓
生成同步报告
    ↓
清理临时资源
    ↓
结束同步
```

##### 股票基本信息同步

- **功能描述**

1. **baostock方案**

- 1. 通过bs.query_stock_basic()接口（不传任何参数）获取全部证券基本信息
- 2. 只保留type=1的股票信息
- 3. 数据组装成下述表格结构
- 4. 基于组装后的数据生成csv文件，csv文件输出目录为{csv文件根目录}/base_stock_info，命名规则为base_stock_info_{yyyyMMdd}.csv，这里的yyyyMMdd是数据采集日期，如果存在同名文件，先删除已存在文件
- 5. 将组装后的数据写入数据表base_stock_info，写库的时候要注意如果数据不存在（使用ts_code判断），执行insert，反之执行update

- **数据来源说明**

| exchange_code | 交易所编码 | 根据code前缀映射交易所编码，sz.=SZSE、sh.=SSE |
| sector_code | 板块编码 | 留空 |
| sector_name | 板块名称 | 留空 |

##### 股票基本面信息同步

- **功能描述**

1. **baostock方案**

- 1. 查询base_stock_info表中未退市的数据，需要获取的字段时ts_code、stock_code和stock_name
- 2. 遍历第1步获取到的数据，通过bs.query_profit_data(code="{ts_code}", year={year}, quarter={quarter})获取前一个季度的股票基本面数据
- 3. baostock接口有QPS=50限制，每50只股票为一组，采集完一组，休眠1秒中，再采集下一组
- 4. bs.query_profit_data接口的参数传递逻辑是
  - **ts_code**：base_stock_info表中获取的ts_code
  - **year**：表示年份
  - **quarter**：表示季度，有效值为[1|2|3|4]，分别表示[1季度｜2季度｜3季度｜4季度]。
  - **传惨逻辑**：默认查询当前年+前一季度的数据，如果没有获取到数据，再查询一次前两季度的数据，如果还没有获取到数据，视为没有该股票的数据
  - **传惨举例**：当前日期为2025年4月19日，第一次查询参数为[year=2025, quarter=1]，如果没有查询到数据，第二次查询参数为[year=2024, quarter=4]，如果还没有查询到数据，视为没有该股票的数据
- 5. 数据组装成下述表格结构组装成下表结构
- 6. 基于组装后的数据生成csv文件，csv文件输出目录为{csv文件根目录}/base_fundamentals_info，命名规则为base_fundamentals_info_{yyyyMMdd}.csv，这里的yyyyMMdd是数据采集日期，如果存在同名文件，先删除已存在文件
- 7. 将组装后的数据写入数据表base_fundamentals_info，写库的时候要注意如果数据不存在（使用ts_code判断），执行insert，反之执行update
- 8. 数据采集时，展示从baostock接口采集数据的进度，格式为：[当前数/总数](股票编码-股票名称) 进度: 进度百分比% baostock: 耗时s
- 9. 记录关键节点总耗时，包括：通过baostock接口采集数据的总耗时，生成csv文件的总耗时，写库总耗时

- **数据来源说明**

| 字段名 | 描述 | baostock方案 |
|--------|------|--------------|
| ts_code | ts编码 | 从base_stock_info表中获取ts_code字段 |
| stock_code | 股票编码 | 从base_stock_info表中获取stock_code字段 |
| stock_name | 股票名称 | 从base_stock_info表中获取stock_name字段 |
| disclosure_date | 信息披露日期 | 通过bs.query_profit_data()获取报告日期，通常为季度末日期 |
| total_share | 总股本 | 通过bs.query_profit_data()获取totalShare字段，单位为股 |
| float_share | 流通股本 | 通过bs.query_profit_data()获取floatShare字段，单位为股 |

##### 日K线数据同步

- **功能描述**

1. **pytdx文件方案**

- 1. 通过pytdx的TdxDailyBarReader.get_df方法加载{通达信数据根目录}/vipdoc/{market}/lday/*.day文件，文件名是股票的不包含"."的ts编码
- 2. 过滤出时间范围内的日k线数据
- 3. 提取出股票ts_code集合，查询base_fundamentals_info表数据，查询出ts_code、stock_code、stock_name和float_share这4个字段
- 4. 如果base_fundamentals_info表中不包含某些股票信息，则提取出这些股票的ts_code集合，查询base_stock_info表数据，查询出ts_code、stock_code和stock_name这3个字段
- 5. 数据组装成下述表格结构组装成下表结构，这里换手率（turnover_rate）字段需要单独计算
- 6. 基于组装后的数据生成csv文件，每个交易日生成一个csv文件，csv文件输出目录为{csv文件根目录}/his_kline_day，命名规则为his_kline_day_{yyyyMMdd}.csv，这里的yyyyMMdd是交易日期，如果存在同名文件，先删除已存在文件
- 7. 将组装后的数据写入数据表his_kline_day，写库的时候要注意如果数据不存在（使用ts_code+trade_date判断），执行insert，反之执行update
- 8. 数据采集时，展示进度，格式为：[当前数/总数](股票编码-股票名称) 进度: 进度百分比% 耗时
- 9. 记录关键节点总耗时，包括：读取day文件总耗时，生成csv文件的总耗时，写库总耗时

通达信数据文件目录结构：
- 北交所股票数据：vipdoc/bj/lday
- 沪市股票数据：vipdoc/sh/lday
- 深证股票数据：vipdoc/sz/lday

| ts_code | TS代码 | 从.day文件名提取，提取规则为：{文件名前2位}+"."+{文件名后6位}，例如文件名为sh000001.day，ts_code为sh.000001 |
| stock_code | 股票编码 | 从.day文件名提取后6位作为股票编码，提取规则为：{文件名后6位}，例如文件名为sh000001.day，stock_code为000001 |
| stock_name | 股票名称 | 从base_fundamentals_info或base_stock_info表中获取stock_name字段，优先base_fundamentals_info表 |

| high | 最高价 | 通过DailyBarReader解析.day文件获取high字段 |
| low | 最低价 | 通过DailyBarReader解析.day文件获取low字段 |

| preclose | 昨日收盘价 | 通过DailyBarReader解析.day文件获取前一天数据的close字段，如果没有获取到，就使用本条数据open字段值 |

### 股票分析功能

#### 立体K线数据生成

**功能描述：**
基于5分钟K线数据，按照价格变动幅度重新聚合K线，生成"立体K线"，突破传统时间维度的K线分析限制。

**生成规则：**
1. **触发条件**：股价相对于当前立体K线的开盘价涨幅达到2.5%
2. **K线生成逻辑**：
   - 每根立体K线代表一次涨幅2.5%的价格变动区间
   - 包含该区间内的：开盘价、最高价、最低价、收盘价
   - 成交量、成交额、换手率为区间内累计值
3. **时间标记**：使用区间结束时间作为K线时间戳

**算法流程：**
```
输入：5分钟K线数据，股票代码，起始日期，结束日期
输出：立体K线序列

步骤：
1. 按时间顺序读取5分钟K线
2. 初始化第一根立体K线：
   - 开盘价 = 第一个5分钟K线的开盘价
   - 当前参考价 = 开盘价
3. 遍历后续5分钟K线：
   a. 计算当前价相对于参考价的涨跌幅
   b. 如果涨幅 ≥ 2.5%：
      - 生成新的立体K线
      - 更新参考价为新K线的收盘价
   c. 否则：累积到当前立体K线
4. 输出所有生成的立体K线
```

**输出字段说明：**

| 字段名 | 描述 | 数据来源 |
|--------|------|----------|
| ts_code | TS代码 | 股票ts_code |
| stock_code | 股票编码 | 股票编码 |
| stock_name | 股票名称 | 股票名称 |
| trade_date | 交易日 | 区间结束时间的K线交易日，yyyyMMdd |
| trade_time | 交易时间 | 区间结束时间的K线交易时间，hhmm |
| open | 开盘价 | 区间开始时间的K线open价格 |
| high | 最高价 | 区间内K线的high价格 |
| low | 最低价 | 区间内K线的low价格 |
| close | 收盘价 | 区间结束时间的K线的close价格 |
| volume | 成交量 | 区间内股票成交量累计值 |
| amount | 成交额 | 区间内股票成交额累计值 |
| change_rate | 涨跌幅 | 区间内股票涨跌幅累计值，理论值应固定为2.5% |
| turnover_rate | 换手率 | 区间内股票累积换手率 |
